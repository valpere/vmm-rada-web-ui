# VMM Rada — User Guide

VMM Rada is a deliberation tool: multiple AI models answer the same
question, evaluate each other's answers, and a Chairman model produces a
synthesized final answer. The goal is a response that is more considered,
more balanced, and less prone to a single model's blind spots than any
individual model could produce alone.

The backend supports seven deliberation strategies (PeerReview, RoleBased,
Majority, GenerateRankRefine, MultiAgentDebate, MixtureOfAgents, Delphi) —
which one runs is a backend configuration choice, not something you pick in
this UI today. This guide describes what you'll see for the default
strategy (PeerReview) and notes where other strategies render differently.

---

## Getting Started

### Prerequisites

- The **Go backend** must be running before you open the interface. By
  default it listens on `http://localhost:8001`.
- The **frontend dev server** must be running: `npm run dev` (starts at
  `http://localhost:5173`).

### Opening the app

Navigate to `http://localhost:5173` in your browser. You will see a sidebar
on the left and a chat area on the right.

## Interface Overview

```
┌─────────────────────┬──────────────────────────────────────────┐
│   Sidebar           │   Chat area                              │
│                     │                                          │
│   + New Conver…     │   Ask a question — multiple models       │
│                     │   answer independently, peer-review      │
│   ● My Question     │   each other, and a chairman synthesises │
│   ○ Another Q       │   the final answer.                      │
│                     │   [suggested prompt chips]                │
└─────────────────────┴──────────────────────────────────────────┘
```

**Sidebar** — lists your saved conversations, newest first. Each entry's
`···` menu offers **Rename** and **Delete** (delete is immediate, no
confirmation prompt). Titles are auto-generated by the backend once the
pipeline finishes; new conversations show "New Conversation" until then.

**Chat area** — when no conversation is active, shows a welcome screen with
a handful of suggested prompts you can click to start immediately.
Otherwise shows the question and the full deliberation response,
progressively, as each stage completes.

---

## Creating a Conversation

1. Click **+ New Conversation** in the sidebar, or click a suggested prompt
   on the welcome screen.
2. Type your question in the text area.
3. Press **Enter** to send, or **Shift+Enter** for a line break within the
   question. Alternatively, click **Send**.

> **One question per conversation.** Each conversation holds exactly one
> question and one response. The input field disappears after you send. To
> ask another question, start a new conversation.

## The Deliberation Pipeline

After you send your question, the interface updates progressively as each
stage completes — you don't need to wait for the whole pipeline to finish.

### Stage 0 — Clarification (optional)

If the backend has clarification enabled and the chairman decides your
question needs more context, a round of free-text questions appears first
(rendered as Markdown). Answer as many as you like — or click **Skip** — and
submit to continue. This can repeat for a few rounds depending on backend
configuration; most deployments have it disabled, in which case Stage 1
starts immediately.

### Stage 1 — Individual Responses

Each model answers your question independently, without seeing what the
others said.

A spinner shows while the models are working. When Stage 1 completes, a
tabbed panel appears:
- Each tab is one model's response, labeled by a shortened model ID (e.g.
  `gpt-5.1`); the full identifier (e.g. `openai/gpt-5.1`) appears above the
  text.
- Responses are rendered as Markdown — headers, lists, code blocks.

**What to look for:** skim a few tabs to see where the models agree and
diverge before reading Stage 2.

### Stage 2 — varies by strategy

What appears here depends on which strategy the backend is running. For the
default **PeerReview** strategy:

Each model re-reads all Stage 1 responses (anonymized as "Response A",
"Response B", …) and ranks them best-to-worst. When it completes, an
accordion expands to:

- **Per-reviewer tabs** — each reviewer's ranked list, shown with real
  model names substituted for the anonymous labels (a display convenience;
  the underlying evaluation used the anonymous labels).
- **Aggregate Rankings** — a combined ranked list (`#1`, `#2`, …) with each
  model's aggregate score, plus a consensus indicator (`W=0.83 strong` —
  higher means the reviewers agreed more).

Other strategies show a different Stage 2 shape entirely — a vote-tally bar
chart (Majority), a ranked candidate list with per-criterion score bars
(GenerateRankRefine), a round-by-round debate timeline
(MultiAgentDebate), stacked proposer/aggregator panels (MixtureOfAgents), a
per-round mean/std-dev/convergence panel (Delphi), or a brief
"roles are complementary" note (RoleBased, which has no peer-ranking step).
See [streaming.md](./streaming.md#stage-2-kind-values) for the full
strategy-to-view mapping.

### Stage 3 — Final Answer

The Chairman model reads the question, all Stage 1 responses, and the
Stage 2 result, then synthesizes one final answer.

- **Chairman: `<model-name>`** — the model that produced this answer.
- Rendered as Markdown.

**If Stage 3 fails** (e.g. quorum wasn't met), an error banner appears
instead of a final answer.

## Conversation History

The sidebar shows all conversations, newest first, with title and
message count. Click any entry to reload it.

**Reloaded conversations lose ephemeral display state.** `label_to_model`
and the strategy-specific Stage 2 metadata are not persisted — a reloaded
`peer_ranking` conversation, for example, shows the original anonymous
labels rather than resolved model names, since the mapping only exists
during the live streaming response.

---

## Tips

**Ask open questions.** The pipeline is most valuable where genuine
disagreement is possible — analysis, comparison, trade-offs. Narrow
factual questions produce less interesting Stage 2 output.

**Check the Chairman's model** in Stage 3 — different chairman models
synthesize differently; some are more conservative, some more opinionated.
This (and which models participate at all) is backend configuration.

**One question = one conversation.** There's no multi-turn follow-up; each
question is a standalone deliberation. Start a new conversation for the
next question.

---

## Limitations

| Limitation | Detail |
|-----------|--------|
| One question per conversation | Input disappears after sending; start a new conversation for the next question |
| No multi-turn conversation | Each question is deliberated from scratch; prior context isn't fed back |
| Reloaded conversations lose ephemeral metadata | Label→model mapping and strategy-specific Stage 2 detail only exist during the live stream |
| No question editing | No way to edit a sent question; start a new conversation |
| No response regeneration | No retry/regenerate button |
| No strategy picker in the UI | Strategy (`council_type`) is currently hardcoded to `"default"` by the frontend; which strategy that resolves to is backend configuration |

## Backend Configuration (for reference)

The backend controls which models participate and which strategy runs.
Commonly relevant variables (full list in the backend's own docs):

| Variable | Purpose |
|----------|---------|
| `RADA_MODELS` | Comma-separated model list for the default PeerReview strategy |
| `CHAIRMAN_MODEL` | Stage 3 synthesizer model |
| `TITLE_MODEL` | Conversation-title generation model (runs concurrently) |
| `DEFAULT_RADA_TYPE` | Which registered strategy `council_type: "default"` resolves to |
| `CLARIFICATION_MAX_ROUNDS` | Enables/limits Stage 0; `0` (default) disables it |
| `PORT` | Backend port (default `8001`) |

The frontend connects to the backend URL set by `VITE_API_BASE` (only
needed when not using the dev-server proxy — see `.env.example`).
